The M × N Integration Problem

Every host re-implements every connector. Every backend re-implements every host plugin.

Same problem these protocols solved

  • HTTP — for browsers and web servers
  • LSP — for editors and language tools
  • ODBC / JDBC — for apps and databases
  • MCP — for LLM apps and their tools/data
  • A2A — for agents collaborating across stacks

Part 1 — MCP

Model Context Protocol

MCP: The Three Roles

Confusing MCP threads always trace back to mixing these up.

RoleWhat it isExamples
HostThe LLM app the user sees. Owns the model + trust.Claude Desktop, VS Code Copilot, ChatGPT
ClientConnector inside the host. 1:1 with a server.The piece of Claude that talks to GitHub MCP
ServerProcess exposing tools / resources / prompts.Your code

The model never speaks MCP — the host translates.

The Five Primitives (+ Roots)

Servers offer any subset. None are mandatory.

PrimitiveWho invokesPurpose
ToolsModel (with host approval)Do things; may have side effects
ResourcesApp / userProvide readable context
PromptsUserReusable templates (slash commands)
SamplingServer → hostServer asks host's LLM to generate
ElicitationServer → userServer asks user for structured input

Roots: host tells server which URI scopes it may touch. Tools/resources/prompts are server primitives; sampling/elicitation are client capabilities.

Transport and Wire Format

JSON-RPC 2.0 with two standardized transports.

Pick one — sometimes both

  • stdio — host spawns server as a subprocess. Local tools. Implicit auth (runs as the user).
  • Streamable HTTP — JSON-RPC over HTTPS to a single endpoint, with SSE for streaming. Remote. Auth via OAuth 2.1.
  • Older two-endpoint "HTTP + SSE" transport exists in early specs — new servers should target Streamable HTTP.

MCP Session Lifecycle

Every server follows the same shape.

Five phases

  • Initialize — exchange protocol version + capabilities
  • List — tools/list, resources/list, prompts/list
  • Invoke — tools/call, resources/read, prompts/get
  • Notify — change notifications, progress updates, cancellations
  • Shutdown — clean close

Implementing an MCP Server

Design first. Code second.

Step 0: Design Before You Code

Write a one-page server design that answers five questions.

Server design questions

  • Who is the user? Developer in an IDE? End user in chat?
  • What system are you wrapping? One bounded context per server.
  • Which primitives? Tools (do), Resources (attach), Prompts (templates).
  • Local or remote? stdio for personal/dev; HTTP for SaaS.
  • What is the trust boundary? Auth, scopes, destructive ops, secrets.

Pick an SDK — Same Pattern Everywhere

Official SDKs: TypeScript, Python, Go, C#/.NET, Java, Kotlin, Swift, Ruby, Rust.

const server = new McpServer({ name: "weather", version: "1.0.0" });

server.tool(
  "get_forecast",
  { city: z.string(), days: z.number().int().min(1).max(7) },
  async ({ city, days }) => fetchForecast(city, days)
);

await server.connect(new StdioServerTransport());

That is the entire server for a small case. The SDK handles JSON-RPC, schema advertisement, and streaming.

Designing a Tool — Three Customer Surfaces

Plus: read vs. write, idempotency, output shape, error contract.

SurfaceWhy it matters
NameThe model reads this to decide whether to call. Verb + noun. Stable forever once published.
DescriptionThe model's only documentation. Write it for an LLM: when to call, what it returns, what it doesn't do.
Input schemaJSON Schema. Validates before your code runs. Be strict; reject extras.

Resources vs. Tools vs. Prompts

Pick the right primitive for the job.

Rules of thumb

  • Tool — the model can call it; side effects OK.
  • Resource — the host can attach it; read-only data with a URI.
  • Prompt — the user triggers it; parameterized template (slash command).
  • "Let the model search my DB" → tool.
  • "Let the user pin a table into chat" → resource.

Auth, Secrets, Multi-Tenancy

Where most MCP servers fail in production.

Non-negotiables

  • stdio: inherit user env. Read keys from env vars or OS keychain.
  • HTTP: OAuth 2.1 is the recommended path. Validate bearer tokens, check the audience, use PKCE.
  • Local HTTP: validate Origin and bind to loopback — defends against DNS rebinding.
  • Never accept secrets as tool arguments — they'd land in chat logs.
  • Derive tenant from the token, never from arguments. The model can be tricked.

Versioning Your Server

Tool schemas are a public API. Users and host configs pin to names.

ChangeOK without major bump?
Add a new toolYes — free
Add an optional parameterYes — free
Add a required parameterNo — major bump
Rename or remove a toolNo — major bump + deprecation period

Observability for MCP Servers

What good operators log and measure.

Log every call

  • Tool name, caller identity, request ID
  • Duration, success / error code
  • Input size (not necessarily content — PII!)
  • Whether the call was a retry

Expose as metrics

  • p50 / p95 / p99 latency per tool
  • Error rate per tool, per host
  • Bytes / tokens returned (host's cost driver)
  • Concurrent sessions

MCP Best Practices

Lessons people learn the hard way.

Tool Design Best Practices

Tool quality drives model behavior more than any prompt you write.

Six guidelines

  • Fewer, sharper tools beat many overlapping ones.
  • Self-documenting names: cancel_subscription_with_refund beats cancel(flag=true).
  • Default to read-only; add writes only with a clear confirmation UX.
  • Return just enough data — paginate big lists.
  • Prefer structured output — JSON the model can index into.
  • Treat tool descriptions as product surface, not afterthoughts.

Security Best Practices

The model is a confused deputy. Treat tool arguments as untrusted input.

Hard rules

  • No path traversal, SQL concat, or shell exec inside a tool.
  • Mark destructive tools clearly — hosts will gate them with confirmation prompts.
  • Honor roots when the host provides them. Refuse paths outside.
  • Never echo secrets back. Redact in resources.
  • Sandbox network access where it makes sense.

Performance and Reliability

Long, blocking tool calls are the #1 source of bad LLM UX.

Engineering checklist

  • Use progress notifications and server-streamed messages for long work.
  • Do not assume token-level streaming of tool results works across all hosts.
  • Implement cancellation so users can stop runaway calls.
  • Set timeouts — per call and per session.
  • Cache resource reads where appropriate.

Testing Your Server

Test before you wire into a real host — or you'll debug in chat.

Four layers

  • MCP Inspector — official click-every-tool debugger.
  • Golden tests — (input → expected output) pairs per tool, in CI.
  • Schema tests — confirm bad input is rejected.
  • Host integration smoke test — at minimum in one real host before each release.

Documentation Has Two Audiences

Write to both. Neither is optional.

For the LLM

  • Tool descriptions
  • Parameter docs
  • Examples in the description
  • Assume model has never seen your product

For humans

  • README with install steps
  • Host config snippet (at least one host)
  • Required scopes and secrets
  • Versioning + deprecation policy

MCP in Claude Code, Copilot, ChatGPT

Same protocol. Three different host stories.

Claude Desktop and Claude Code

The first widely-deployed MCP hosts.

Host-side specifics

  • Config: claude_desktop_config.json (desktop) or per-project .mcp.json (Claude Code).
  • Transports: stdio first-class; remote Streamable HTTP with OAuth supported.
  • Scopes (Claude Code): user / project / local — controls who sees the server.
  • UX: tools → model-callable; resources → attachment menu; prompts → slash commands.
  • Destructive tools prompt the user before running.

GitHub Copilot (VS Code, JetBrains, CLI)

MCP is how Copilot agent mode reaches outside the IDE.

Host-side specifics

  • Config: workspace or user mcp.json; curated registry with one-click installs.
  • Transports: stdio and Streamable HTTP — host brokers OAuth flows.
  • Tools show up in agent mode; user can toggle individual tools per chat.
  • Workspace-trust gating + per-tool confirmation prompts for non-allowlisted tools.
  • Copilot coding agent and Copilot CLI can reuse MCP server configurations, but supported transports and capabilities differ per surface.

ChatGPT

MCP via "connectors" — remote only.

Host-side specifics

  • Register the connector against a remote MCP server URL with OAuth.
  • Streamable HTTP only — no stdio in a hosted product.
  • Surfaces as connectors users enable per conversation and as Deep Research data sources.
  • Implication: ChatGPT-compatible servers must be publicly hosted + OAuth-capable.

The Portability Promise — and Caveats

One server, three hosts — in theory.

Where portability gets messy

  • Auth is the hardest portability problem — each host's OAuth conventions differ.
  • Tool UX varies — confirmation, streaming, and resource rendering all host-specific.
  • Capability support varies — sampling and elicitation are not universal.
  • Design for the lowest-common-denominator host you care about, then progressively enhance.

Interlude — MCP Apps

When tools need a UI.

What Are MCP Apps?

MCP servers can ship interactive UI — not just text and JSON.

Two threads of the same story

  • ChatGPT Apps (Apps SDK) — Oct 2025, first widely-shipped implementation. Built on MCP.
  • MCP UI / "Apps" — community standardization of the same pattern across hosts.
  • The model still drives the conversation. The component is what the user sees and clicks.
  • Host support is moving fast — assume the matrix has changed; verify before relying on it.

How an MCP App Is Wired Up

Reuse all of MCP. Add three things.

Three additions

  • A UI resource — HTML at ui://component/... with MIME type text/html;profile=mcp-app.
  • A tool that references it — _meta.ui.resourceUri on the tool result. (ChatGPT also accepts the legacy alias _meta['openai/outputTemplate'].)
  • A host ↔ component bridge — sandboxed iframe gets a ui/* JSON-RPC API (and window.openai in ChatGPT) to read structured content, call tools, request follow-up turns, persist state.

Designing MCP Apps Well

Same discipline as good MCP tools, plus front-end realities.

Five guidelines

  • Pick the right surface — add UI when output (chart, map) or input (form, picker, confirmation) is clunky as text.
  • Degrade gracefully — always return useful structuredContent too, so non-Apps hosts still get a sensible answer.
  • Thin client — no business logic in the iframe. Call back to MCP tools for data and writes.
  • Mind the network boundary — outbound calls governed by CSP and declared domains.
  • Version your UI URIs — cache-bust on each release (hosts cache aggressively).

MCP Apps vs. Plain Tools

Pick the right tool for the job.

QuestionPlain toolMCP App
User has to do something (approve, pick, filter)?AwkwardNatural
Model can chain the output to the next tool?YesYes — via structuredContent
Portable across all MCP hosts?YesOnly on Apps-rendering hosts
Right for charts / maps / forms / dashboards?NoYes

Rule of thumb: data the model reasons over → plain tool. Something the user sees or interacts with → MCP App.

MCP Apps Trade-offs

Be honest about the cost.

Three honest caveats

  • Portability is uneven today — host support is moving fast; verify current support.
  • You're now also a front-end engineer — components need design, accessibility, CSP, dependency hygiene.
  • Auth and tenancy still live on the server — the model and the component are both clients in your trust model.

Part 3 — A2A

Agent-to-Agent Protocol

Why A2A?

MCP connects LLMs to tools. A2A connects agents to other agents.

The scenario

  • You run a travel agent (LangChain), an expense agent (Copilot Studio), a calendar agent (vendor).
  • You want a fourth, executive-assistant agent, to coordinate all three.
  • Without a protocol → three custom integrations.
  • With A2A → each advertises an Agent Card; any client can discover and delegate.
  • Introduced by Google (Apr 2025). Governed by the Linux Foundation since mid-2025.

A2A Is Opaque About Internals

This is what makes cross-vendor delegation safe.

The calling agent never sees the called agent's:

  • Prompts
  • Model choice
  • Internal tools
  • Memory
  • It only sees the contract: skills, inputs, outputs, lifecycle.

A2A Core Concepts

Memorize the vocabulary. Every A2A discussion uses it.

ConceptMeaning
Agent CardJSON at /.well-known/agent-card.json. The "menu" other agents read.
SkillNamed capability — coarse-grained, often multi-turn.
TaskUnit of work. Lifecycle: submitted → working → input-required → completed / failed / canceled. Also: auth-required, rejected.
Message / PartA turn; carries text, file, or structured data parts.
ArtifactDurable output of a task (PDF, patch, dataset).
Push notificationOptional webhook for async progress updates.

A2A Transport — Multiple Bindings

Transport-agnostic by design. Pick a binding.

Bindings + extras

  • JSON-RPC 2.0 over HTTP(S)
  • gRPC binding
  • HTTP+JSON / REST binding
  • Server-Sent Events layered for streaming responses
  • Webhooks for push notifications on long-running tasks
  • Auth: whatever HTTP supports (OAuth, API keys, mTLS) — declared in the Agent Card

A2A Collaboration Lifecycle

Six phases from discovery to artifact.

Lifecycle

  • Discover — fetch the Agent Card.
  • Authenticate — per scheme declared in the card.
  • Send task — message/send or message/stream.
  • Iterate — agent may go input-required; client (or its user) responds.
  • Receive artifacts — durable outputs.
  • Close — completed / failed / canceled.

Implementing an A2A Agent

Server side, then client side.

Server Side: Expose Your Agent

Make an existing agent reachable to other agents.

Six steps

  • Write the Agent Card — pick which skills to expose, not every capability.
  • Pick modes — text? files? structured JSON? streaming?
  • Implement endpoints for your binding — JSON-RPC message/send, message/stream, tasks/get, tasks/cancel; or REST POST /message:send, GET /tasks/{id}; or gRPC SendMessage, GetTask. Use an SDK.
  • Choose auth — API keys (internal) or OAuth / JWT (cross-org).
  • Implement push notifications for long-running tasks.
  • Serve the card at /.well-known/agent-card.json.

Client Side: Call Other Agents

Make your agent capable of delegating.

Five concerns

  • Maintain a registry of known agents — list, catalog, or discovery service.
  • Fetch and cache Agent Cards. Refresh periodically.
  • Treat remote agents as a tool type in your planner — often surfaced as an MCP tool.
  • Handle input-required gracefully — bubble to user or answer from context.
  • Propagate user identity carefully — OBO tokens when acting on the user's behalf.

Design Decisions That Matter

Get these wrong early and they hurt forever.

Three trade-offs

  • Coarse vs. fine skills — "plan a trip" (A2A skill), not "look up an airport code" (MCP tool).
  • Sync vs. async — long tasks need streaming + push notifications from day one.
  • State boundary — what lives inside the remote agent vs. what the client passes every call.

A2A Best Practices

Trust, hygiene, operations.

Agent Card Hygiene

The card is documentation a delegating LLM reads.

Three rules

  • Write skills for an LLM reader — clear, specific, no marketing fluff.
  • Don't lie. If you can't really do X, don't list X — real requests will get routed to you.
  • Version the card. Bump on breaking changes; clients can pin.

Safety and Trust

Prompt-injection now crosses agent boundaries.

Hard rules

  • Authenticate every request — never trust a caller's identity claim alone.
  • Re-authorize on the user's behalf — check the human can do this, not just that the caller is "trusted".
  • End-to-end task correlation IDs — cross-agent debugging is impossible without them.
  • Treat agent responses as untrusted input — sanitize before feeding back to your model.

Operational Concerns

Multi-tenant agents are SaaS. Treat them that way.

Three controls

  • Per-skill quotas and rate limits — a misbehaving caller can DDoS via task creation.
  • Cancellation must actually cancel — don't keep burning tokens.
  • Surface cost — return cost metadata in artifacts so callers can budget.

MCP vs. A2A at a Glance

Different problems, same era, designed to coexist.

AspectMCPA2A
ConnectsLLM app ↔ toolsAgent ↔ agent
GranularityFine (one function)Coarse (multi-turn skill)
CallerModel inside a hostAnother agent client
StateMostly stateless per callStateful tasks with lifecycles
DiscoveryList APIs at session startAgent Card at well-known URL

How They Stack

MCP is how an agent uses tools. A2A is how an agent uses another agent.

[ User ]
   |
[ Host application + LLM ]
   |  speaks MCP to tools
   +--> [ MCP server: filesystem ]
   +--> [ MCP server: postgres  ]
   +--> [ MCP server: "delegate to agent" ]
            |  speaks A2A
            +--> [ Remote agent: travel   ]
            +--> [ Remote agent: expenses ]
                    \__ uses MCP internally for its own tools

A common 2025-era architecture: A2A across agents, MCP inside each agent.

Which Protocol Do I Build?

Ask in this order.

Decision flow

  • Tool/data source for hosts you don't own? → MCP server.
  • Self-contained agent for other agents to call? → A2A agent.
  • Building the host yourself? → Consume both.
  • One-shot internal feature for a single LLM app? → Maybe neither yet.
  • Use protocols when you need portability, decoupling, or third-party reach.

Week 9 Takeaways

LLM apps are orchestrators of tools and agents — not monoliths.

  • MCP = LLM ↔ tools/data. Small set of well-defined primitives.
  • A2A = agent ↔ agent. Coarse skills, explicit task lifecycle.
  • The hard parts are auth, safety, and the UX of confirmation.
  • Tool descriptions and Agent Cards are product surface, not afterthoughts.
  • Pin the protocol version — both specs evolved rapidly through 2024–2025.
  • Use SDKs. Use the Inspector. Test before shipping.
